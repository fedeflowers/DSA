# Explanation of the LeetCode Solution for the "Tree Node" Problem

## 1. Brief Explanation of the Approach

The SQL query provided categorizes nodes in a tree structure into three types: "Root", "Inner", and "Leaf". 

- **Root Nodes**: These are nodes that do not have a parent node (i.e., `p_id` is `NULL`).
- **Inner Nodes**: These are nodes that have children (i.e., they are referenced as a parent in the `p_id` of other nodes) and also have a parent (i.e., their own `p_id` is not `NULL`).
- **Leaf Nodes**: These are nodes that do not have any children (i.e., no other node references them as a parent in `p_id`) and have a parent node.

The solution utilizes Common Table Expressions (CTEs) to build subqueries for each type of node:

1. **CTE `r`** identifies the root nodes.
2. **CTE `i`** identifies the inner nodes by joining the tree with itself.
3. **CTE `l`** identifies the leaf nodes that are not root nodes, ensuring that there is no overlap by excluding root nodes.

Finally, the results from each of the CTEs are combined using `UNION` to produce a comprehensive list of all nodes with their respective types.

## 2. Time and Space Complexity Analysis

### Time Complexity
- The time complexity is O(N), where N is the number of nodes in the `tree` table. This is because each query scans the `tree` table at most a couple of times (once for each CTE), making the overall complexity proportional to the number of nodes.

### Space Complexity
- The space complexity is also O(N) due to the storage requirements for the result sets created by the CTEs. The intermediate tables essentially create temporary storage for the nodes categorized by their types.

## 3. Why This Approach is Efficient

- **Clear Categorization**: The use of CTEs allows for a clear and organized structure to differentiate between the types of nodes in the tree. Each CTE focuses on a specific problem area, which enhances code readability and maintainability.
  
- **Single Pass for Each Category**: The structure employs a straightforward method of evaluating each category in separate passes, avoiding complex condition checks and subqueries. Each CTE effectively operates like a single scan, which is advantageous for performance.

- **Use of SQL UNION**: The `UNION` operator efficiently combines the results from the three different CTEs, allowing for a simplified final result set without excessive joining or filtering.

In conclusion, this SQL solution effectively utilizes relational database principles to achieve efficient categorization of nodes in a hierarchical structure, providing clear and maintainable code.

WITH r AS (
    SELECT id, 'Root' AS type FROM tree WHERE p_id IS NULL
),
i AS (
    SELECT t1.id, 'Inner' AS type
    FROM tree t1 JOIN tree t2 ON t1.id = t2.p_id
    WHERE t1.p_id IS NOT NULL
),
l AS (
    SELECT t1.id, 'Leaf' AS type
    FROM tree t1 LEFT JOIN tree t2 ON t1.id = t2.p_id
    WHERE t2.id IS NULL AND t1.p_id IS NOT NULL  -- ← guards both root and leaf overlap
)
SELECT * FROM r UNION SELECT * FROM i UNION SELECT * FROM l
